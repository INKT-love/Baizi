# 白子完整手机控制设计

## 目标

让白子在 Android 上以 Agent 方式完成与 Operit 同级别的手机自动化：理解聊天指令、读取当前界面、操作应用与系统、执行多步工作流，并把结果回传至对话。配置面向普通用户：优先引导 Shizuku，Root 作为高权限补充，无障碍服务承担跨应用界面自动化。

这是 Android 专属功能。桌面端与 iOS 保持现有行为，不显示不可用的控制入口。

## 范围与系统边界

完整能力包括：

- 界面自动化：截图、UI 树读取、按文字或控件定位、点击、长按、输入、滑动、返回。
- 系统与应用控制：启动或停止应用、主页、最近任务、通知栏、音量、亮度、网络相关开关、通知和剪贴板。
- 文件与命令：文件读取、写入、移动、复制、压缩、解压；Shell 命令、应用安装与卸载管理。
- 自动任务：将多步操作保存为工作流，支持定时或条件触发、暂停、继续、取消和执行历史。
- 对话反馈：每项操作在聊天内显示进度、结果和可恢复的错误说明，模型据此决定下一步。

Android 的锁屏、生物识别、支付验证、系统安全确认页和厂商受保护界面不能被绕过。遇到这些页面时，任务暂停并提示用户接手；用户完成后可继续执行。

## 总体架构

新增 Android 原生“手机控制中枢”，Flutter 负责界面、数据模型与对话工具适配。模型不直接获得 Shell 或系统权限；所有调用必须通过中枢校验、执行并返回结构化结果。

```text
聊天模型
  <-> 结构化手机工具调用与结果
Flutter 手机控制服务
  <-> MethodChannel/EventChannel
Android 手机控制中枢
  |- 权限与能力检测
  |- 通道选择、风险分级、确认策略
  |- 任务队列、取消、超时、重试与日志
  |- UI 自动化、截图、系统、文件、Shell、工作流适配器
  |
  +-> Shizuku (优先) -> Root (补充) -> 无障碍服务 (界面操作)
```

通道按能力而非单一全局开关选择：系统或 Shell 操作优先 Shizuku，不可用或能力不足时才尝试 Root；读取和操作第三方应用可见界面依赖无障碍服务。任务开始前先检测所需能力，不能满足时直接返回可读错误和相应的“去连接”入口。

## 原生组件

### PhoneControlBridge

Flutter 与 Android 的唯一桥接边界。提供：读取授权状态、发起授权引导、执行工具、取消任务、订阅任务状态、读取执行摘要和管理工作流。桥接层只传递结构化参数和结果，不能传递任意 Android 对象。

### PhoneControlCoordinator

执行中枢。它维护单个前台执行队列，避免多个 AI 工具并发争抢界面；执行前校验参数、能力、风险与确认策略；执行中发布步骤进度；执行后标准化输出。每个步骤有超时和有限重试，连续失败或检测到受保护页面时停止后续步骤。

### CapabilityAdapters

- `ShizukuAdapter`：检测 Shizuku 服务和用户授权，执行适合该通道的 Binder/Shell 操作。
- `RootAdapter`：检测 `su` 可用性，为 Shizuku 不可用或需要 Root 的动作提供受控 Shell 执行。
- `AccessibilityAdapter`：通过无障碍服务读取活动窗口与节点树，完成控件定位、点击、输入、滚动和系统导航。
- `ScreenAdapter`：在具备 Android 允许的权限和用户授权时提供屏幕观察能力，向模型返回必要的压缩截图或结构化描述。
- `FileAppAdapter`：封装文件、包管理、通知、剪贴板与系统动作，按能力中枢选择通道。

适配器只负责一种能力，不能自行决定确认策略或向聊天写入内容。

### Tool Registry

注册对模型可见的手机工具，并为每个工具声明参数模式、需要的能力、风险等级、默认超时和结果模式。工具类别包括：

- 观察：`get_ui_tree`、`capture_screen`、`find_element`、`get_device_state`。
- 界面：`tap`、`long_press`、`input_text`、`scroll`、`back`、`home`、`open_notifications`。
- 系统和应用：`launch_app`、`stop_app`、`list_apps`、`set_volume`、`set_brightness`、系统状态开关、通知与剪贴板操作。
- 文件与命令：受限路径的文件操作、压缩解压、包管理和 Shell 命令。
- 工作流：创建、执行、暂停、恢复、取消和读取历史。

工具结果统一包含 `status`、`summary`、`details`、`next_action` 和可选 `artifact`。聊天只显示摘要，详细 UI 树、截图和命令输出按需展开。

## 首次配置与设置

设置页新增“手机控制”，默认只显示一个状态卡和“开始配置”。配置向导固定顺序如下：

1. 检查 Shizuku，说明其为推荐方式；已运行时打开 Shizuku 授权，不存在或未运行时给出简短启动说明与重新检查按钮。
2. 当 Shizuku 未可用或无法满足所需能力时，显示 Root 可用状态和连接入口；不要求普通用户理解 Root 命令。
3. 引导开启无障碍服务，明确它用于读取可见界面、点击、输入和滑动。
4. 选择确认策略。默认“按风险确认”；可改为“全部确认”或“全部允许”。

高级功能包含能力诊断、工作流管理、日志、截图和 UI 树保留设置、各通道重新连接及总停用开关。功能入口在 Android 聊天页和设置中可见；未配置时入口只会跳转配置向导。

## 确认策略与风险

确认策略由用户选择并保存在本地：

- `按风险确认`：低风险观察和可逆导航直接执行；会改变数据、系统状态、发送内容、安装或卸载应用、删除文件等高风险动作需逐项确认。
- `全部确认`：任何会改变手机状态的动作均逐项确认；纯观察可直接执行。
- `全部允许`：启用时显示一次明确确认。确认后，包括高风险动作在内的所有已注册手机工具均无需再次弹窗。

“全部允许”不是绕过 Android 自身的安全边界。它启用后，聊天执行卡片仍显示进度和停止按钮；设置中可立即切回其他策略，切换后对后续步骤生效。受保护页面、授权缺失、目标控件不确定或连续失败时仍会暂停。

## 执行体验和恢复

AI 发起任务后，聊天中出现紧凑执行卡：当前步骤、已完成数量和停止按钮。任务结束后显示结果摘要；用户可展开查看每步详情。模型收到结果后可继续调用下一项工具。

处理规则：

- 通道断开：重新检测，允许时自动回退到另一通道；否则结束并提供恢复路径。
- UI 改变：重新读取 UI 树并重新定位一次；仍不匹配时停止，避免误操作。
- 超时和连续失败：结束任务，记录失败步骤和原因。
- 受保护页面：暂停任务，不尝试绕过；用户完成手动步骤后可恢复。
- 用户停止：立即取消队列中未开始的步骤并停止可取消的当前动作，向模型回传已完成部分。

执行摘要、授权状态、确认策略和工作流保存在本地。截图、UI 树和完整 Shell 输出默认仅用于当前任务，不长期保存；用户可在高级功能中显式保留。敏感文本应在日志展示前脱敏。

## 测试与验收

- 单元测试：风险分级、通道选择、参数校验、工具结果转换、任务状态机和日志脱敏。
- Flutter 测试：配置向导状态、策略切换、工具卡片、停止/恢复入口、Android 以外平台的隐藏行为。
- Android 集成测试：Shizuku 已授权和未授权、Root 可用与不可用、无障碍启用与关闭、通道回退、前后台切换与进程重启。
- 设备验收：常见 Android 版本与至少一种厂商系统；验证界面定位、输入、滚动、应用控制、文件操作、工作流以及失败暂停。
- 回归：现有聊天、模型选择、角色卡、经期关怀和后台任务不得受影响。

## 交付拆分

完整目标分阶段交付，阶段间保持同一工具协议：

1. 控制中枢、Shizuku/Root 检测、无障碍服务、配置向导、确认策略和执行卡片。
2. UI 树、截图、基础界面自动化和系统/应用控制。
3. 文件、包管理、Shell、通知、剪贴板与高级系统能力。
4. 工作流、定时和条件触发、执行历史及恢复。
5. 多设备兼容性、稳定性、性能与完整回归验证。

每阶段均可构建可用 APK，但最终发布前必须完成全部阶段和设备验收。
